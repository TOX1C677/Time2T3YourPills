ВВЕДЕНИЕ
Данный отчёт описывает разработку мобильного приложения Time2T3YourPills для платформы Android на языке Java, предназначенного для управления приёмом лекарственных препаратов и напоминаний о времени их приёма.
Приложение разработано для помощи пользователям в соблюдении режима приёма медикаментов, обеспечивая автоматические уведомления, управление списком лекарств, настройку таймеров обратного отсчёта и ведение информации о пациентах. Работа выполнена в рамках дипломного проекта с использованием современных компонентов Android Architecture Components, паттерна MVVM и локальной базы данных Room для надёжного хранения данных. Приложение предоставляет интуитивно понятный интерфейс с нижней навигацией, фоновый сервис для работы таймера, систему уведомлений с возможностью подтверждения приёма и отложенного напоминания, а также функционал для управления данными пациентов с поддержкой заметок.

АРХИТЕКТУРА СИСТЕМЫ
Архитектура мобильного приложения Time2T3YourPills построена на многоуровневом подходе с использованием паттерна Model-View-ViewModel (MVVM), что обеспечивает разделение ответственности между компонентами и упрощает тестирование и поддержку кода. Модель данных представлена классами Medication, User и Note, которые инкапсулируют информацию о лекарственных препаратах, пациентах и связанных с ними заметках. Эти классы используют аннотации Room для определения структуры таблиц базы данных с поддержкой внешних ключей и каскадного удаления.
Слой доступа к данным реализован через интерфейсы DAO (Data Access Object): MedicationDao, UserDao и NoteDao, которые предоставляют методы для синхронных и асинхронных операций с базой данных с использованием LiveData для реактивного обновления пользовательского интерфейса. Центральным элементом работы с данными служит класс AppDatabase, построенный на базе Room Database версии 2.5.2, обеспечивающий единую точку доступа к локальной SQLite базе данных с версионированием схемы и поддержкой миграций.
Слой репозиториев представлен классами MedicationRepository и UserRepository, которые абстрагируют работу с базой данных и предоставляют единый интерфейс для ViewModel. Репозитории используют AsyncTask и Executor для выполнения операций ввода-вывода в фоновых потоках, что предотвращает блокировку основного потока пользовательского интерфейса. ViewModel слой включает MedicationViewModel для управления данными о лекарствах, UserViewModel для работы с информацией о пациентах и SharedViewModel для обмена данными между фрагментами, что позволяет сохранять состояние при поворотах экрана и изменениях конфигурации.
Представление реализовано на базе фрагментов Android: TimerFragment для отображения и управления таймером обратного отсчёта, MedicationFragment для работы со списком лекарств, PatientFragment для управления данными пациентов и AboutFragment для отображения информации о приложении. Главная активность MainActivity организует навигацию между фрагментами через нижнюю панель навигации (Bottom Navigation) и обеспечивает связь с фоновым сервисом TimerService через механизм ServiceConnection. Для коммуникации между компонентами используется LocalBroadcastManager для передачи обновлений таймера и ViewModel для обмена данными между фрагментами.
Фоновый сервис TimerService реализует логику работы таймера обратного отсчёта на базе CountDownTimer, обеспечивая непрерывную работу даже при сворачивании приложения. Сервис интегрирован с системой уведомлений через класс NotificationHelper, который создаёт каналы уведомлений для Android 8.0 и выше и формирует уведомления с действиями подтверждения и отложенного напоминания. Обработка действий пользователя в уведомлениях реализована через BroadcastReceiver: ConfirmReceiver для подтверждения приёма лекарства и SnoozeReceiver для отложенного напоминания.

ФУНКЦИОНАЛЬНОСТЬ СИСТЕМЫ
Базовый функционал приложения
В базовой версии приложение предоставляет комплексный набор возможностей для управления приёмом лекарств. К основным функциям относятся:
•	Управление списком лекарственных препаратов с возможностью добавления, редактирования и просмотра информации о каждом медикаменте, включая название, дозировку, интервал между приёмами в минутах и количество повторений.
•	Настройка таймера обратного отсчёта на основе параметров выбранного лекарства с автоматическим запуском при добавлении нового препарата и отображением оставшегося времени в реальном времени.
•	Система уведомлений о времени приёма лекарства с высоким приоритетом, обеспечивающая своевременное информирование пользователя даже при заблокированном экране устройства.
•	Управление данными пациентов с возможностью сохранения имени, отчества и заметок, что позволяет вести учёт информации о нескольких пользователях приложения.
•	Нижняя навигация между основными разделами приложения (Таймер, Лекарства, Пациенты, О приложении) для быстрого доступа к функциональности.
•	Локальное хранение данных в базе данных Room с автоматическим сохранением всех изменений и восстановлением состояния при перезапуске приложения.
Эти компоненты формируют основу приложения, обеспечивая базовую функциональность для управления приёмом лекарств.

Новый функционал, реализованный в период разработки
2.1 Фоновый сервис для работы таймера
Для обеспечения непрерывной работы таймера даже при закрытии приложения был реализован фоновый сервис TimerService, который работает независимо от жизненного цикла активности и фрагментов. Сервис использует CountDownTimer для обратного отсчёта времени и отправляет обновления в пользовательский интерфейс через LocalBroadcastManager, что позволяет отображать актуальное состояние таймера в реальном времени. При истечении времени таймер автоматически отправляет уведомление и, при наличии настроенных повторений, перезапускается для следующего цикла. Сервис поддерживает связь с активностью через механизм ServiceConnection, что обеспечивает двустороннюю коммуникацию между UI и фоновым процессом.

2.2 Система уведомлений с действиями
Была разработана система уведомлений на базе NotificationHelper, которая создаёт каналы уведомлений для Android 8.0 и выше и формирует уведомления с действиями пользователя. Каждое уведомление содержит кнопку "Подтвердить", при нажатии на которую таймер перезапускается, и кнопку "Отложить" для переноса напоминания. Уведомления имеют высокий приоритет и настраиваются для работы даже в режиме "Не беспокоить", что критично для медицинских напоминаний. Обработка действий реализована через специализированные BroadcastReceiver, которые передают команды в TimerService для выполнения соответствующих операций.

2.3 Реактивное обновление пользовательского интерфейса
Для обеспечения отзывчивости интерфейса и автоматического обновления данных при изменениях в базе данных была внедрена система реактивного программирования на базе LiveData. Все операции чтения из базы данных возвращают LiveData объекты, которые автоматически уведомляют подписанные компоненты об изменениях, что позволяет обновлять списки лекарств, информацию о пациентах и состояние таймера без необходимости ручного обновления. Это обеспечивает консистентность данных между различными фрагментами приложения и упрощает синхронизацию состояния.

2.4 Управление жизненным циклом через ViewModel
Для сохранения данных при поворотах экрана и изменениях конфигурации устройства была реализована система ViewModel, которая хранит данные, связанные с пользовательским интерфейсом, независимо от жизненного цикла активности и фрагментов. MedicationViewModel управляет данными о лекарствах, UserViewModel отвечает за информацию о пациентах, а SharedViewModel обеспечивает обмен данными между фрагментами, в частности, передачу настроек таймера от MedicationFragment к TimerFragment. Это устраняет необходимость повторной загрузки данных при пересоздании компонентов и улучшает производительность приложения.

2.5 Репозиторный паттерн для абстракции данных
Для обеспечения слабой связанности между слоями приложения и упрощения тестирования был внедрён репозиторный паттерн через классы MedicationRepository и UserRepository. Репозитории инкапсулируют всю логику работы с базой данных, включая асинхронные операции вставки, обновления и удаления, и предоставляют единый интерфейс для ViewModel. Это позволяет в будущем легко заменить источник данных, например, добавить синхронизацию с облачным хранилищем, без изменения кода ViewModel и UI компонентов. Репозитории используют Executor и AsyncTask для выполнения операций в фоновых потоках, предотвращая блокировку основного потока.

2.6 Структурированное хранение данных в Room Database
Была реализована локальная база данных на базе Room Database с тремя основными таблицами: medication_table для хранения информации о лекарствах, user_table для данных пациентов и note_table для заметок, связанных с пациентами. База данных использует внешние ключи для обеспечения целостности данных и каскадное удаление для автоматической очистки связанных записей. Версионирование схемы базы данных позволяет выполнять миграции при изменении структуры таблиц, а использование LiveData в запросах обеспечивает автоматическое обновление UI при изменениях данных.

2.7 Адаптер для отображения списка лекарств
Для эффективного отображения списка лекарств в MedicationFragment был разработан MedicationAdapter на базе RecyclerView.Adapter, который обеспечивает производительное отображение больших списков через механизм переиспользования представлений. Адаптер получает обновления через LiveData и автоматически обновляет отображаемый список при изменениях в базе данных, что обеспечивает синхронизацию данных без необходимости ручного обновления интерфейса.

2.8 Обработка разрешений для уведомлений
Для обеспечения работы уведомлений на Android 13 и выше была реализована система запроса разрешения POST_NOTIFICATIONS через манифест приложения. Это позволяет приложению корректно работать на всех версиях Android и обеспечивает пользователю контроль над уведомлениями через системные настройки.

2.9 Разработка пользовательского интерфейса
В настоящее время ведётся разработка пользовательского интерфейса приложения на базе Material Design Components, адаптированного под особенности различных категорий пользователей, включая пожилых людей и людей с ограниченными возможностями. Планируется использование компонентов BottomNavigationView для навигации, RecyclerView для списков, ConstraintLayout для гибкой верстки и стандартных элементов Material Design для кнопок и полей ввода. Особое внимание уделяется внедрению увеличенных элементов управления, контрастных цветовых схем для лучшей читаемости, крупных шрифтов и упрощённой навигации. Также планируется реализация поддержки тёмной темы через отдельный файл themes.xml в папке values-night для улучшения комфорта использования в условиях низкой освещённости. Дизайн будет ориентирован на простоту использования и минимизацию количества действий, необходимых для выполнения основных операций, что критично для медицинских приложений, где важна скорость и точность взаимодействия. Данная работа находится на стадии проектирования и реализации.
