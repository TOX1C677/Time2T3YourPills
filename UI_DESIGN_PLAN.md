# План проектирования экранов Time2T3YourPills

**Назначение:** пошаговый план работы над визуальным дизайном клиентского приложения от первых набросков до готовых макетов и токенов темы для Flutter.  
**Контекст:** продукт — только клиент пациента (см. `FLUTTER_MIGRATION_PLAN.md`), аудитория — люди с паркинсонизмом, требования к UI зафиксированы в `DESIGN_GUIDELINES_PARKINSON.md`.  
**Результат этапа:** утверждённый набор макетов в Figma + `design_tokens.json` / `lib/app/theme/` + обновлённые ссылки из `DESIGN_GUIDELINES_PARKINSON.md` на экраны.

---

## 0. Принципы работы над дизайном

1. **Один экран — один основной фокус** (п. 5 гайдлайна). Если экран начинает «обрастать» — разбиваем на шаги мастера, а не на вкладки внутри.
2. **Сначала состояния, потом красота.** Для каждого экрана сразу рисуем все ключевые состояния (ждёт / активно / ошибка / пусто / офлайн), а не только «идеальный» кейс.
3. **Доступность — критерий приёмки, а не «полируем потом».** Каждый макет проверяется по чеклисту из §9 `DESIGN_GUIDELINES_PARKINSON.md` **до** передачи в Flutter.
4. **Цветом смысл не несём в одиночку.** Всегда: цвет + иконка + слово.
5. **Макеты проверяются на двух размерах шрифта:** базовый (18 sp) и крупный (24 sp) — чтобы поймать переполнение текста заранее.
6. **Макеты проверяются в двух темах:** светлой и тёмной, обе — на AA минимум, основной текст — AAA.

## 1. Карта экранов (scope первой версии)

Экраны, которые входят в проект дипломной работы. Всё, что не в списке — в бэклог.

| № | Экран | Назначение | Критичность |
|---|-------|------------|-------------|
| 1 | **Онбординг** | 3 экрана: приветствие → запрос разрешений → первый препарат | высокая |
| 2 | **Логин / регистрация** | вход в аккаунт, синхронизация с сервером | высокая |
| 3 | **Сегодня (главный)** | ближайший приём + лента дня | **ключевой** |
| 4 | **Уведомление о приёме** | системный push + full-screen intent | **ключевой** |
| 5 | **Режим «Большая кнопка»** | полноэкранный приём одним тапом (для «off»-периодов) | высокая |
| 6 | **Лекарства — список** | все препараты пациента | высокая |
| 7 | **Лекарства — мастер добавления/редактирования** (6 шагов) | создание/правка препарата | высокая |
| 8 | **История приёмов** | лог по дням + экспорт PDF | средняя |
| 9 | **Помощь / FAQ** | офлайн-объяснения, SOS-кнопка | средняя |
| 10 | **Профиль и настройки** | шрифт, тема, звук, контакт помощника | средняя |
| 11 | **Пустые состояния и ошибки** | «нет лекарств», «нет сети», «ошибка сервера» | высокая |

**Состояния, которые рисуются для каждого функционального экрана:**

- `loading` — скелетон / индикатор
- `empty` — пусто (первый запуск, нет данных)
- `content` — обычный режим
- `offline` — баннер «работаем из кэша» + сам контент
- `error` — ошибка загрузки с кнопкой «Повторить»

### Этап B. Дизайн-токены и визуальная система (2–3 дня)

Токены должны быть собраны **до отрисовки экранов**, иначе макеты расползутся.

#### B.1. Палитра

| Токен | Light | Dark | Назначение |
|-------|-------|------|------------|
| `color.bg.screen` | `#F4F1EC` | `#0E1013` | фон экрана |
| `color.bg.card` | `#FFFFFF` | `#1A1D22` | фон карточки |
| `color.text.primary` | `#16181D` | `#F2EFE8` | основной текст (≥ AAA) |
| `color.text.secondary` | `#3E4450` | `#B7BCC7` | вторичный текст (≥ AA) |
| `color.border.default` | `#D9D3C7` | `#2B2F37` | рамки карточек, разделители |
| `color.accent.positive` | `#0B6E4F` | `#4FD39A` | «Принято», «Пора пить» |
| `color.accent.warning` | `#5A4A17` | `#E0B84A` | «Ждёт», отложено |
| `color.accent.negative` | `#B23A1F` | `#FF7A5C` | «Пропущено», ошибка |
| `color.focus.ring` | `#0B6E4F` 3 dp | `#4FD39A` 3 dp | клавиатурный фокус и «пора пить» |

Каждая пара проверяется инструментом контраста (например, [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)) — результаты фиксируются в таблице.

#### B.2. Типографика

| Токен | Размер | Weight | Использование |
|-------|--------|--------|---------------|
| `text.display` | 96 sp | 600 | таймер на главном |
| `text.display.xl` | 160 sp | 600 | таймер в режиме «Большая кнопка» |
| `text.title.l` | 28 sp | 600 | заголовок карточки активного приёма |
| `text.title.m` | 22 sp | 600 | названия лекарств, заголовки секций |
| `text.body` | 20 sp | 400 | основной текст |
| `text.caption` | 18 sp | 400 | подписи, вторичный текст |

Шрифт: **Atkinson Hyperlegible** (основной) + `Inter` fallback. Межстрочный 1.4, `fontFeatures: tabular-nums` для таймера и времени.

#### B.3. Размеры и отступы

| Токен | Значение | Использование |
|-------|----------|---------------|
| `size.touch.min` | 64 dp | минимум для любой кнопки |
| `size.touch.primary` | 88 dp | главная кнопка действия |
| `size.touch.fullscreen` | 40 % высоты экрана | кнопка «Принял» в режиме «Большая кнопка» |
| `space.xs` | 4 dp | внутри чипа |
| `space.s` | 8 dp | между однородными элементами |
| `space.m` | 16 dp | базовый отступ |
| `space.l` | 24 dp | между секциями |
| `space.xl` | 32 dp | вокруг критичных кнопок |
| `radius.card` | 20 dp | карточки |
| `radius.button` | 16 dp | кнопки |
| `border.width` | 1.5 dp | рамка карточки |
| `border.width.focus` | 3 dp | состояние «пора пить» и клавиатурный фокус |

#### B.4. Иконки

- Стиль: линейный, толщина 2.5 dp, контейнер 40–48 dp.
- Набор: `pill`, `clock`, `check`, `cross`, `snooze`, `bell`, `palm` (режим «Большая кнопка»), `share`, `phone`, `settings`.
- **Правило:** иконка никогда не стоит без подписи (п. 2.4, 5.2 гайдлайна).

#### B.5. Звук и тактилька

| Событие | Звук | Вибрация |
|---------|------|----------|
| Напоминание | двойной тёплый «бим-бом» | двойная короткая |
| Подтверждение приёма | короткий «дзинь» | одиночная |
| Пропуск / ошибка | низкий «бу» | длинная |

Всё отключаемо раздельно в настройках.

**Выход этапа B:**

- `design/tokens/design_tokens.json` — машинно-читаемые токены.
- `design/tokens/README.md` — обоснование каждого значения со ссылкой на пункт гайдлайна.
- Figma-страница **«Foundations»** с цветами, типографикой, спейсингом, иконками.

### Этап C. Компоненты (Figma UI Kit) (2–3 дня)

Собираем **библиотеку компонентов** в Figma до экранов. Иначе каждый экран рисуется «с нуля» и теряется единообразие.

Минимальный набор:

- [ ] `Button / Primary` — 88 dp, полная ширина, состояния: default / pressed / disabled / loading.
- [ ] `Button / Secondary` — 72 dp, состояния те же.
- [ ] `Button / Ghost` — текстовая, 64 dp.
- [ ] `Card / Default` — фон + рамка 1.5 dp + padding 20 dp.
- [ ] `Card / Active` (пора пить) — рамка 3 dp акцентного цвета.
- [ ] `Card / Negative` (опоздание) — рамка 3 dp негативного цвета.
- [ ] `Chip / Time` — 56 dp, например `08:00`, состояния selected / default.
- [ ] `ListItem / Intake` — строка истории приёма (✓/✕/○ + время + название).
- [ ] `Toggle` — 56 dp высотой, с текстом состояния справа.
- [ ] `Stepper` — `[−15]  14:00  [+15]`, кнопки 64 dp.
- [ ] `Banner / Offline` — полная ширина, 56 dp, жёлтый нейтральный.
- [ ] `Banner / Error` — полная ширина, 56 dp, негативный цвет.
- [ ] `BottomNav` — 5 вкладок, иконка + подпись 18 sp, активная — жирнее + полоска сверху.
- [ ] `AppBar` — заголовок 22 sp, справа до 2 действий ≥ 48 dp.
- [ ] `Dialog / Confirm` — короткий текст + две равные кнопки.

Каждый компонент — с **auto-layout** и вариантами (тема × состояние × размер шрифта).

**Выход:** Figma-страница **«Components»** + таблица соответствия «компонент → Flutter-виджет» в `design/components.md` (для этапа E).

### Этап D. Экраны и состояния (5–7 дней)

Порядок отрисовки — **от самого критичного**, чтобы раньше показать пользователям/научруку:

1. **Сегодня (главный)** — 4 состояния карточки приёма (ждёт / пора / опаздывает / принято) + 2 темы × 2 размера шрифта = 16 артбордов.
2. **Уведомление** — Android heads-up, Android полноэкранный, iOS alert, iOS lock screen. 4 артборда.
3. **Режим «Большая кнопка»** — 2 темы.
4. **Мастер добавления лекарства** — 6 шагов × 2 темы = 12 артбордов. Для шага «Название» отдельно состояние «Пусто / С подсказками / С автодополнением».
5. **Список лекарств** — пусто / 1 препарат / 3+ препарата / пауза у одного препарата.
6. **История** — сегодня / 7 дней / 30 дней / пусто.
7. **Профиль и настройки** — основной вид + изменение размера шрифта (превью).
8. **Помощь / FAQ** — свёрнутый список + раскрытая карточка + SOS-кнопка.
9. **Онбординг** — 3 экрана.
10. **Логин / регистрация** — вход / регистрация / ошибка сети / ошибка сервера.
11. **Экраны ошибок и офлайн** — баннер в контексте экрана + отдельный экран «Нет связи с сервером».

**Правила внутри этапа D:**

- Каждый экран размещается на сетке **360×800 dp** (Android baseline) и дублируется на **390×844 dp** (iPhone 13/14).
- Сразу же рядом — **вариант с шрифтом 24 sp**: если текст не умещается, правим макет, а не шрифт.
- **Тёмная тема** — не «инверсия», а отдельный артборд. Проверяется контраст.
- На каждом артборде подписаны **ссылки на пункты** `DESIGN_GUIDELINES_PARKINSON.md`, которые он реализует (например: «кнопка 88 dp — §1; текст 20 sp — §2.1; цвет + иконка + слово — §3.2»).

**Выход:** Figma-страница **«Screens»**, структурированная по номерам из §1 карты экранов.

### Этап E. User flows (1–2 дня)

Схема переходов между экранами — чтобы видеть «дыры» до разработки.

Минимум 4 flow:

1. **Первый запуск:** Онбординг → Разрешения → Логин → Мастер первого препарата → Сегодня.
2. **Ежедневный приём:** Push-уведомление → Принял (из шторки) **ИЛИ** открыть приложение → Сегодня → Принял.
3. **Редактирование расписания:** Сегодня → BottomNav «Лекарства» → карточка → «Изменить» → мастер → сохранить → назад.
4. **Офлайн-сценарий:** потеря сети → баннер → принять приём → восстановление сети → отправка отложенной записи (если в ТЗ включено редактирование офлайн).

Нарисовать в Figma / FigJam / Miro — один документ `design/flows.md` + ссылка на доску.

**Выход:** `design/flows.md` + экспорт PNG/SVG каждого flow.

### Этап F. Доступность и приёмка (1–2 дня)

До передачи в разработку — **независимая проверка**:

- [ ] Контраст всех пар «текст/фон» проверен инструментом, результаты — в таблице `design/a11y.md`.
- [ ] Все интерактивные зоны ≥ 64 dp (замер прямо в Figma плагином Measure или Stark).
- [ ] Экран выдерживает системный шрифт «Очень крупный» (≈ 200 %) — нет обрезания, горизонтального скролла, наложения.
- [ ] Ни один смысл не передаётся **только** цветом (ручная проверка: макет в ч/б → читается ли статус?).
- [ ] Анимации ≤ 250 мс, альтернативный режим «уменьшить анимацию» описан.
- [ ] В уведомлении действия **совпадают** по смыслу с карточкой в приложении.
- [ ] Режим «Большая кнопка» достижим с главного экрана в 1 тап.
- [ ] SOS-кнопка «Позвонить помощнику» доступна с экрана «Помощь» в 1 тап.

Все пункты — галочки в `design/a11y.md` со скриншотом-пруфом.

**Выход:** `design/a11y.md` + подпись в `DESIGN_GUIDELINES_PARKINSON.md` в разделе «отклонения» (если есть).

### Этап G. Передача в разработку Flutter (1–2 дня)

Цель — чтобы разработчик **не придумывал ничего сам**.

- [ ] Экспортировать токены из Figma в `design/tokens/design_tokens.json` (плагин Tokens Studio или вручную).
- [ ] Сгенерировать Dart-константы в `lib/app/theme/`:
  - `app_colors.dart` — light / dark ColorScheme;
  - `app_typography.dart` — TextTheme из токенов;
  - `app_sizes.dart` — `minTouchTarget`, `primaryButtonHeight`, `cardRadius`, `spaceM` и т. д.;
  - `app_theme.dart` — собранный `ThemeData` для Material 3.
- [ ] Подключить шрифт Atkinson Hyperlegible через `pubspec.yaml`.
- [ ] Описать соответствие «компонент Figma → Flutter-виджет» в `design/components.md` (например: `Button / Primary` → `PrimaryButton` в `lib/shared/widgets/buttons/primary_button.dart`).
- [ ] Ссылки на экраны Figma добавить в конец `DESIGN_GUIDELINES_PARKINSON.md` (как предусмотрено последним абзацем документа).
- [ ] Обновить `FLUTTER_MIGRATION_PLAN.md` §4: заменить «Figma: цветовая система, типографика…» на ссылку на готовый UI-kit.

**Выход:** PR-ready код темы + обновлённые документы + ссылка на Figma с правами «view».

## 5. Что сознательно отложено в бэклог

Эти экраны и фичи **не входят** в текущий объём проектирования — чтобы не размывать фокус диплома:

- Камера / OCR упаковки лекарства.
- Социальные функции, лента, чаты.
- Геймификация: стрики, достижения, очки.
- Глубокая аналитика приёмов (графики, корреляции со сном и т. п.).
- Интеграция с Google Fit / Apple Health.
- Экран врача / роли «лечащий врач» (явно исключено в `FLUTTER_MIGRATION_PLAN.md` §0).
- Мультиязычность UI (на первом этапе — только русский).
- Режим сопровождающего с удалённым управлением расписанием.

Каждое отложенное решение, если вернётся, требует **нового раунда** этапов A–G для добавленных экранов.
