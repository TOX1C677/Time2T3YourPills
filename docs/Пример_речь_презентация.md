**Слайд 2. Актуальность работы**

В образовательных организациях активно используются цифровые инструменты: электронные журналы, расписания, личные кабинеты студентов и преподавателей. Но часть внутренних процессов до сих пор часто ведется вручную или через обычные таблицы.

Одним из таких процессов является бронирование аудиторий. Даже если основное расписание уже составлено, в течение семестра регулярно появляются дополнительные занятия, пересдачи, консультации, собрания и другие мероприятия. Для них нужно быстро найти свободную аудиторию и не создать конфликт с уже существующим расписанием.

Поэтому автоматизация такого процесса является актуальной задачей. Она помогает сделать работу с аудиторным фондом более удобной, наглядной и менее зависимой от ручной проверки таблиц.

**Слайд 3-4. Проблема текущего процесса**

На этом слайде показан пример текущего способа учета занятости аудиторий через Excel-таблицу.

Такой вариант понятен и привычен, но при постоянной работе с ним появляются проблемы. Таблица становится большой и визуально перегруженной. Чтобы найти свободную аудиторию, приходится вручную просматривать строки, столбцы, дни недели и временные интервалы.

Кроме того, при ручном заполнении есть риск ошибки: можно случайно занять уже занятый слот, неправильно учесть числитель или знаменатель, либо не заметить пересечение по времени.

В результате Excel подходит для первичного учета, но не обеспечивает удобной автоматической проверки и быстрого поиска свободных аудиторий.

**Слайд 6. Требования и ограничения системы**

Разрабатываемая система должна позволять пользователю выбирать аудиторию, выбирать учебную неделю, просматривать занятость по временным слотам и создавать новые бронирования.

В приложении поддерживаются два типа бронирований. Регулярное бронирование создается для повторяющихся мероприятий и привязывается к дню недели, номеру пары и типу недели. Разовое бронирование создается на конкретную календарную дату и заданный временной интервал.

Также в системе учитывается чередование недель: числитель, знаменатель и обе недели. Это важно, потому что одна и та же аудитория может быть занята в одну неделю и свободна в другую.

При этом у системы есть ограничения. Она не составляет основное расписание с нуля и не позволяет его редактировать. Основное расписание используется как готовый набор данных. Пользователь работает только с дополнительными бронированиями, которые создаются в свободное время.

**Слайд 7. Средства реализации**

Для разработки клиентской части были использованы React, TypeScript и Vite.

React - это библиотека для создания пользовательских интерфейсов. Она позволяет разделить интерфейс на отдельные компоненты: список аудиторий, выбор недели, таблицу занятости и форму бронирования.

TypeScript - это язык программирования, расширяющий JavaScript за счет строгой типизации. Он помогает заранее описывать структуру данных и снижает вероятность ошибок при разработке.

Vite - это инструмент сборки и запуска клиентского приложения. Он используется для быстрой разработки React-проекта.

Серверная часть была реализована на Node.js и Express. Node.js позволяет выполнять JavaScript-код на сервере, а Express используется для создания маршрутов REST API.

REST API нужен для обмена данными между клиентской и серверной частями. Данные передаются в формате JSON. В этой версии проекта JSON-файлы также используются как хранилище данных: в них находятся аудитории, преподаватели, временные слоты, расписание, бронирования и нерабочие дни.

**Слайд 8. Архитектура разработанной системы**

На этом слайде представлена архитектура разработанной системы.

Пользователь работает с приложением через браузер. В браузере запускается клиентская часть, реализованная на React и TypeScript. Она отображает интерфейс, таблицу занятости, список аудиторий и форму бронирования.

Когда приложению нужны данные, клиентская часть отправляет HTTP-запросы к серверу через REST API. Серверная часть, реализованная на Node.js и Express, принимает запрос, обращается к нужному JSON-файлу и возвращает ответ клиенту.

Например, для отображения таблицы система получает список аудиторий, временные слоты, основное расписание и список созданных бронирований. После этого клиентская часть формирует таблицу занятости и показывает ее пользователю.

Такая архитектура разделяет интерфейс, серверную обработку и хранение данных, поэтому проект проще сопровождать и развивать.

**Слайд 9. Реализация интерфейса**

На этом слайде показан основной экран разработанного приложения.

Слева расположен список аудиторий. Пользователь выбирает нужную аудиторию, после чего таблица занятости перестраивается под выбранное помещение.

В верхней части расположен переключатель учебной недели. Он позволяет переходить между неделями и смотреть занятость в нужном периоде.

Основная часть экрана - таблица занятости. В строках отображаются временные интервалы учебных пар, а в столбцах - дни недели с понедельника по субботу. Занятые ячейки выделяются цветом.

В системе используются разные типы занятости: занятие из основного расписания, регулярное бронирование (желтое) и разовое бронирование (синее). Благодаря цветовому различию пользователь может быстро понять, почему аудитория занята.

Ниже таблицы находится форма создания нового бронирования. В ней выбирается тип бронирования, преподаватель, день недели или конкретная дата, время и дополнительный комментарий.

**Слайд 10. Логика бронирования и тестирование**

Важной частью системы стала логика определения занятости аудитории.

При построении таблицы система проверяет три источника данных: основное расписание, регулярные бронирования и разовые бронирования. Для каждой ячейки таблицы определяется, свободна аудитория или занята.

Для регулярных бронирований выполняется проверка конфликта. Система сравнивает аудиторию, день недели, номер пары и тип недели. Если уже существует бронирование с пересекающимися параметрами, новая запись не создается.

Для разовых бронирований используется календарная дата и временной интервал. Также реализован запрет бронирования на воскресенье, потому что учебная неделя в системе рассматривается с понедельника по субботу.

После реализации было проведено тестирование основных сценариев: загрузка данных, выбор аудитории, переключение недели, отображение расписания, создание регулярного бронирования, создание разового бронирования, проверка конфликта и обновление таблицы после добавления записи.

По результатам проверки основные функции системы работают корректно.
