# План расширения: API, БД, роли, история приёмов, уведомления опекуну

Документ описывает пошаговую реализацию идеи «солидного» дипломного про поверх текущего клиента (`time2t3_your_pills`): домашний сервер на Ubuntu с белым IP и доменом, REST API, PostgreSQL, авторизация по почте, роли «пациент» / «врач или родственник», несколько пациентов у одного опекуна, синхронизация лекарств и истории приёмов, напоминания опекуну о пропуске.

---

## 1. Цели и границы MVP+

| Цель | Критерий готовности |
|------|---------------------|
| Данные не только на устройстве | Лекарства, профиль, история приёмов читаются/пишутся через API с авторизацией |
| Два типа пользователя | После входа — разный набор экранов и прав (пациент vs опекун) |
| Один опекун — несколько пациентов | Привязка «опекун ↔ пациенты», переключатель активного пациента на экране таблеток |
| История приёмов | Запись при подтверждении приёма (время, препарат, доза); просмотр по ролям |
| Пропуск приёма | По таймауту после due — событие/уведомление опекуну «пациент не принял — проверьте» |

**Вне первой итерации (опционально):** полноценный офлайн-first с конфликтами, push на iOS без Apple Developer платного аккаунта, видеозвонки, чат.

## 2. Архитектура (высокоуровнево)

```
[Flutter] ←HTTPS→ [Nginx: TLS, rate-limit] → [API: FastAPI или Node/Nest]
                                              ↓
                                         [PostgreSQL]
                                              ↓
                    [Фоновые задачи: cron/worker — пропуски, напоминания]
```

- **Клиент:** уже есть `dio`, `flutter_secure_storage`, `go_router`, `provider` — достаточно для токенов, interceptors и отдельного «слоя API».
- **Сервер:** один VPS/домашний Ubuntu; **Nginx** как reverse proxy + **Let's Encrypt** (certbot) на вашем домене.
- **БД:** **PostgreSQL** (нормализация, JSON при необходимости для гибких полей препарата).
- **Авторизация:** access + refresh JWT в secure storage; пароль или одноразовый код на почту — см. раздел 5.

### 2.1. Обоснование стека API и БД (что выбрано в репозитории и почему)

| Слой | Выбор | Зачем |
|------|--------|--------|
| **HTTP API** | **Python + FastAPI** | Быстрое описание эндпоинтов, встроенная валидация через **Pydantic**, автоматическая **OpenAPI/Swagger** (`/docs`) — удобно для диплома и демонстрации контракта. Альтернативы (Django REST, Flask, Node/Nest) тяжелее по объёму шаблонного кода или другой экосистеме при уже выбранном Flutter-клиенте на Dart. |
| **ORM** | **SQLAlchemy 2.x** | Один способ описать таблицы и связи в коде; переносимость между окружениями; при росте проекта — миграции через **Alembic**. |
| **БД (прод)** | **PostgreSQL** | Надёжный прод-стек: ограничения целостности, индексы, **JSON/JSONB** для гибких полей (например слоты приёма), при необходимости — сложные отчёты и фоновые задачи без ограничений SQLite. Развёртывание локально/на сервере — через **`db/docker-compose.yml`**. |
| **БД (разработка и CI)** | **SQLite** (файл или in-memory в тестах) | Тесты API в CI **без обязательного Docker** с Postgres: быстрый прогон `pytest`. Локально API по умолчанию может писать в файл SQLite в каталоге `api/` — не нужен поднятый контейнер для первых шагов. |
| **Пароли** | **bcrypt** | Медленный хеш паролей, привычная практика; хранится только хеш в `users.password_hash`. |
| **Сессии мобильного клиента** | **JWT** (access + refresh, HS256 в v1) | Flutter — не браузер с «удобными» httpOnly-cookie; токены в **secure storage** + заголовок `Authorization` естественно стыкуются с **Dio**. Refresh позволяет не заставлять пользователя логиниться при каждом коротком access TTL. |
| **ASGI-сервер** | **Uvicorn** | Стандартный процесс для запуска FastAPI в dev и за reverse proxy (Nginx) в проде. |

**Итог:** один репозиторий с папками **`api/`** (приложение и тесты) и **`db/`** (инфраструктура Postgres и заметки по миграциям); целевая «серьёзная» связка для защиты — **FastAPI + PostgreSQL + JWT**, при этом разработка и CI остаются лёгкими за счёт SQLite там, где полноценный Postgres не обязателен.

## 3. Инфраструктура на Ubuntu (домашний сервер)

### 3.1. Базовая безопасность

- Открыть наружу только **443** (и при необходимости **80** для ACME challenge).
- SSH по ключу, `fail2ban`, отключить root-login по паролю.
- Регулярные обновления `unattended-upgrades` для security-патчей.

### 3.2. Стек API (рекомендация для диплома)

| Вариант | Плюсы |
|---------|--------|
| **Python + FastAPI + SQLAlchemy + Alembic** | Быстро описать модели, автодок Swagger, понятно в пояснительной записке |
| Node + NestJS + TypeORM | Если комфортнее JS/TS |

Дальше в плане зафиксирован **FastAPI + Postgres** как наиболее прямолинейный для «пятерки» по документации.

### 3.3. Nginx

- Проксирование `https://api.ваш-домен.ru` → `uvicorn` на `127.0.0.1:8000`.
- Заголовки `X-Forwarded-For`, таймауты, лимит размера тела запроса.

### 3.4. CI/CD (минимум для диплома)

- Скрипт деплоя: `git pull` + `alembic upgrade head` + `systemctl restart time2t3-api`.
- В записке — блок «развертывание».

## 4. Модель данных (PostgreSQL)

### 4.1. Пользователи и роли

```text
users
  id (uuid, pk)
  email (unique, not null)
  password_hash (nullable, если только magic-link)
  role (enum: patient | caregiver)
  display_name
  email_verified_at
  created_at, updated_at

caregiver_patient_links
  id (uuid)
  caregiver_user_id → users
  patient_user_id → users
  created_at
  optional: nickname ("Мама"), archived_at

-- уникальность пары (caregiver, patient)
```

**Правило:** пользователь с `role=patient` не может быть «опекуном» в смысле UI без отдельной политики; проще — **второй аккаунт** «родственник» с `role=caregiver`, привязка к пациенту через **постоянный код привязки** (см. 4.3).

*Альтернатива «один email — две роли»:* усложняет UX и модель; для диплома лучше **один email — одна роль**, при необходимости два аккаунта на одну почту не давать (unique email).

### 4.2. Профиль пациента (серверный)

```text
patient_profiles
  user_id (pk, fk → users)  -- только для role=patient
  first_name, middle_name
  timezone (IANA string)
  link_token text NOT NULL  -- см. 4.3: один раз при создании пациента, без UPDATE в приложении
  updated_at
```

Синхронизация с текущим `PatientProfile` / `PatientController`. Поле `link_token` на клиенте в локальный профиль можно не дублировать — только отображение по запросу к API.

### 4.3. Привязка пациент ↔ опекун: постоянный код (без email-приглашений)

**Идея:** у каждого аккаунта пациента при регистрации **один раз** генерируется неизменяемый **код привязки** (достаточной длины, напр. 32+ символа в base32/base58 без похожих символов). Код **не перегенерируется** (нет смены/ротации в v1). Опекун вводит этот код у себя в приложении — сервер создаёт запись в `caregiver_patient_links`.

**Хранение на сервере:** колонка `patient_profiles.link_token` (см. §4.2) — значение генерируется при создании записи пациента, **запрет UPDATE** на поле (триггер БД или только слой приложения). Отдельная таблица не обязательна.
- В записке указать: в промышленной версии — хранить только **хеш** кода и показывать plaintext **один раз** при регистрации; для UX «копировать когда угодно из профиля» в учебном проекте допустимо хранение секрета в БД при строгом HTTPS и ограниченном доступе к БД.

**Альтернатива без хранения plaintext:** при регистрации сервер один раз отдаёт код в теле ответа, клиент сохраняет в Keychain/secure storage и может показать в диалоге из локального хранилища; на другом устройстве пациента код не восстановить без механизма «экспорт/бэкап» — для диплома проще колонка на сервере.

**Поток UX**

| Роль | Экран | Действие |
|------|--------|----------|
| **Пациент** | Профиль | Кнопка **«Добавить лечащего врача / родственника»** → модальное окно (bottom sheet / `AlertDialog`): текст-инструкция («покажите или отправьте код опекуну»), **поле с кодом** (read-only или `SelectableText`), кнопка **«Копировать»** (`Clipboard.setData`). Код подгружается с API `GET .../invite-code` при открытии диалога. |
| **Опекун** | Профиль | Кнопка **«Добавить пациента»** → диалог: **поле ввода кода**, кнопка «Привязать» → `POST /caregiver/link-patient` с телом `{ "token": "..." }` → при успехе пациент появляется в `GET /caregiver/patients` и в селекторе на таблетках. |

**Ошибки и края:** неверный код (404/400), попытка привязать уже привязанного пациента (409), пациент ввёл свой же код (запрет self-link), слишком много попыток с одного IP/аккаунта (rate limit).

**Никаких писем для привязки** в этой схеме не требуется: канал передачи кода — лично, мессенджер, звонок (как в реальной жизни «скажи мне код из приложения»).

### 4.4. Лекарства (привязка к пациенту)

```text
medications
  id (uuid)
  patient_user_id → users
  name, dosage
  reminder_mode (enum: interval | scheduled)
  interval_minutes (nullable)
  slot_times (text[] или jsonb: ["08:00","20:00"])
  created_at, updated_at, deleted_at (soft delete)
```

Соответствие текущей модели `Medication` в `lib/core/models/medication.dart`.

### 4.5. История приёмов (intake log)

```text
intake_events
  id (uuid)
  patient_user_id
  medication_id (nullable, если препарат удалили — хранить snapshot)
  medication_name_snapshot, dosage_snapshot
  scheduled_at (когда должны были принять)
  recorded_at (когда нажали «принял»)
  source (enum: patient_app | caregiver_override | system)
  status (enum: confirmed | missed | snoozed)
  snooze_until (nullable)
```

- При **«Подтвердить»** в приложении: `POST /patients/me/intake-events` с `medication_id`, `scheduled_at`, `recorded_at`, `status=confirmed`.
- При **«+15 мин»**: можно записать `snoozed` + `snooze_until` или не писать в историю до финального подтверждения — зафиксировать в ТЗ одно поведение (рекомендация: писать **только подтверждённый приём** в «историю лекарств», снуз — в отдельной таблице `intake_snoozes` при желании).

### 4.6. События «не принял» (для опекуна)

```text
missed_intake_alerts
  id, patient_user_id, medication_id, due_at, detected_at
  notified_caregiver_at (nullable)
```

Worker раз в N минут сравнивает `next_due` / расписание с фактом `intake_events` и создаёт алерт.

## 5. Авторизация по почте

### 5.1. Варианты (выбрать один для v1)

| Подход | Реализация | Комментарий |
|--------|------------|-------------|
| **Пароль + подтверждение email** | Регистрация → письмо с токеном → `/verify` | Классика для диплома |
| **Magic link** | Вход без пароля, ссылка одноразовая | Удобно, нужен надежный SMTP |
| **OTP 6 цифр** | Код в письме, короткий TTL | Хорошо для пожилых без запоминания пароля |

**SMTP:** отдельный почтовый ящик (Yandex/SendGrid/Mailgun) или локальный relay только для dev; в проде — внешний провайдер, чтобы письма не улетали в спам.

### 5.2. Токены

- **Access JWT** (короткий TTL, 15–60 мин).
- **Refresh JWT** (длиннее, хранение в httpOnly cookie *или* только в `flutter_secure_storage` для мобильного клиента — для Flutter обычно пара токенов в secure storage).
- Interceptor в `dio`: 401 → refresh → повтор запроса.

### 5.3. Роль после регистрации

- Экран **«Кто вы?»**: «Пациент» / «Врач или родственник» → `POST /auth/register` с полем `role`.
- Запретить смену роли без поддержки (или админ-флаг в БД для демо).

## 6. REST API (черновой контракт)

Префикс: `https://api.ваш-домен.ru/v1`.

### Auth

- `POST /auth/register` — email, password, role, display_name  
- `POST /auth/login`  
- `POST /auth/refresh`  
- `POST /auth/verify-email`  
- `POST /auth/logout` (инвалидация refresh по jti)

### Пациент (Bearer, role=patient)

- `GET/PATCH /patients/me/profile`
- `GET/POST/PATCH/DELETE /patients/me/medications`
- `POST /patients/me/intake-events` — подтверждение приёма
- `GET /patients/me/intake-events?from=&to=` — история
- `GET /patients/me/invite-code` — **неизменяемый** код привязки для показа пациенту в диалоге «Добавить врача/родственника» (генерируется на сервере при регистрации аккаунта с `role=patient`, в ответе только для владельца аккаунта).

### Опекун (role=caregiver)

- `POST /caregiver/link-patient` — тело `{ "token": "<код от пациента>" }` → валидация, создание `caregiver_patient_links`, идемпотентность при повторной отправке той же пары (уже есть связь → 200 без дубликата).
- `GET /caregiver/patients` — список привязанных пациентов (id, имя)
- `GET /caregiver/patients/{id}/medications` — полный CRUD как у пациента, но с `{id}`
- `GET /caregiver/patients/{id}/intake-events` — история выбранного пациента

### Служебное (опционально)

- `GET /health` — для мониторинга

**Версионирование:** `/v1` обязательно для эволюции API в записке.

## 7. Изменения во Flutter-приложении

### 7.1. Слои данных (соответствие текущему коду)

| Сейчас | Станет |

**Конфигурация:** `--dart-define=API_BASE_URL=https://api...` для release-сборки.

### 7.2. Маршрутизация (`app_router.dart`)

- Цепочка: **Splash** → проверка токена → **Login/Register** → **Role onboarding** (если новый) → **Shell**.
- **Shell для пациента:** текущий `AppShell` — таймер, таблетки, профиль.
- **Shell для опекуна:** упрощённый нижний бар или drawer: **«Пациенты / Таблетки»**, **«История»**, **«Профиль / выход»** (без таймера у опекуна, если не нужен — или read-only таймер «следующий приём выбранного пациента» как опция).

### 7.3. Привязка: UI пациента и опекуна

- **Пациент, `profile_screen` (или отдельный блок «Опекуны»):** кнопка **«Добавить лечащего врача / родственника»** → `showModalBottomSheet` / `showDialog` с текстом инструкции, отображением кода (`FutureBuilder` / запрос `GET /patients/me/invite-code`), кнопка **«Копировать»** (`Clipboard`), при желании **«Поделиться»** (`Share.share` на мобильных).
- **Опекун, профиль:** кнопка **«Добавить пациента»** → диалог с **`TextField`** для кода (trim, без пробелов в середине по желанию), **«Привязать»** → `POST /caregiver/link-patient` → `SnackBar` успех/ошибка → обновить список пациентов в `CaregiverPatientsController`.
- Код **никогда не запрашивать «сгенерировать заново»** — в UI нет такой кнопки; на сервере нет эндпоинта ротации в v1.

### 7.4. Выбор пациента (опекун)

- В `MedicationsScreen` (или обёртка `CaregiverMedicationsScope`): **Dropdown / Segmented** в `AppBar` или под ним: список из `GET /caregiver/patients`.
- Выбранный `patientId` в `Provider` / `CaregiverSessionController`.
- Все запросы лекарств с префиксом `/caregiver/patients/{id}/...`.

### 7.5. Синхронизация лекарств

- При открытии списка: `GET` → merge в локальную БД/GetStorage (как сейчас) для офлайн-просмотра последней версии.
- После редактирования: `PATCH` → обновить локально.
- **Конфликт:** last-write-wins по `updated_at` на v1 достаточно для диплома.

### 7.6. Таймер и уведомления (пациент)

- `IntakeTimerController` после подтверждения: локально как сейчас **+** `POST intake-events`.
- При ошибке сети: очередь outbox (у вас уже есть `OutboxRepository` / `outbox_entry` — расширить типами «intake_event»).

### 7.7. История приёмов — UI

**Универсальное место для кнопки «История»:**

- Вынести в **AppBar действие** (иконка часовая стрелка) на экранах, где история релевантна: **Профиль** (пациент), **Таблетки** (и пациент, и опекун с контекстом выбранного пациента).
- Либо одна пункт **«История приёмов»** в меню «⋯» рядом с «О приложении» на профиле **и** дублировать в AppBar таблеток для опекуна.

**Экран `IntakeHistoryScreen`:**

- Фильтры: период (7 дней / 30 дней / всё).
- Список: дата-время, название, доза, статус.
- Для опекуна — заголовок с именем выбранного пациента.

## 8. Уведомления опекуну о пропуске

### 8.1. Определение «пропуск»

- Через **T минут** после `due_at` (например 30–60, настраиваемо на сервере или в профиле пациента) нет записи `intake_events` со `status=confirmed` для этой пары (medication + scheduled_at).
- Worker помечает `missed_intake_alerts` и шлёт уведомление опекунам.

### 8.2. Канал доставки

| Канал | Реализация |
|-------|------------|
| **Email** | Проще всего на домашнем сервере: шаблон письма + SMTP |
| **Push FCM** | Нужны ключи Firebase, настройка Android; iOS — сертификаты Apple |
| **In-app badge** | Опекун при открытии видит список алертов `GET /caregiver/alerts` |

**Рекомендация v1:** email + in-app список; push — как улучшение в записке «перспектива».

### 8.3. Спам-контроль

- Не более одного алерта на ту же `(patient, medication, due_at)` или агрегация в сутки.

## 9. Безопасность и соответствие здравоохранению (легко описать в записке)

- Только HTTPS; заголовки безопасности (Nginx).
- Хеш паролей: **argon2** или **bcrypt**.
- Авторизация на каждом запросе; опекун не может читать чужих пациентов без `caregiver_patient_links`.
- **Код привязки:** высокая энтропия (≥128 бит случайных данных), константное время сравнения при проверке; **rate limit** на `POST /caregiver/link-patient` (и при желании на `GET /patients/me/invite-code`) против перебора.
- Логи без ПДн в plain text (email маскировать); **не** логировать полный `link_token`.
- Политика хранения: срок хранения истории (например 1 год) — опционально cron-очистка.

## 10. Этапы реализации (roadmap)

### Фаза A — Сервер и БД (1–2 недели)

1. Репозиторий `time2t3-api`: FastAPI, конфиг из env, подключение Postgres.

## 11. Риски и упрощения

| Риск | Митигация |
|------|-----------|
| Домашний IP сменится | Динамический DNS (ddclient) + мониторинг |
| Письма в спам | Внешний SMTP, SPF/DKIM для домена |
| Сложность офлайн | v1: «нужна сеть для входа», кэш только для чтения последней синхронизации |
| Два человека правят одно лекарство | `updated_at` + сообщение пользователю при конфликте |
