**ВВЕДЕНИЕ**

В ходе научно-исследовательской работы была выполнена разработка веб-системы бронирования аудиторий для учебного подразделения. Необходимость такой системы связана с тем, что учет занятости аудиторий часто ведется вручную в табличных файлах, что усложняет поиск свободного времени и повышает вероятность ошибок при добавлении новых записей.

Основная задача работы заключалась в создании приложения, которое позволяет отображать существующее учебное расписание, просматривать занятость аудиторий и создавать дополнительные регулярные и разовые бронирования. Отдельное внимание было уделено учету учебных недель по типу числитель/знаменатель, проверке конфликтов и удобному представлению данных в интерфейсе.

В результате была сформирована клиент-серверная система, включающая клиентскую часть на React и TypeScript, серверную часть на Node.js и Express, REST API и хранение данных в JSON-файлах. Разработанное решение может использоваться как основа для дальнейшего развития модуля бронирования в информационной системе факультета.

**1\. АНАЛИЗ ИСХОДНОГО СОСТОЯНИЯ БРОНИРОВАНИЯ**

На начальном этапе был рассмотрен текущий способ учета занятости аудиторий. Основным инструментом выступала Excel-таблица, в которой аудитории располагались по столбцам, а дни недели и временные интервалы учебных пар - по строкам. Занятость отмечалась цветовой заливкой и текстовыми пометками.

Такой подход является привычным и достаточно простым для первичного учета, однако при постоянной работе имеет заметные ограничения. Пользователю приходится вручную просматривать большое количество ячеек, учитывать тип недели, сверять занятость с уже внесенными данными и самостоятельно контролировать возможные пересечения.

После анализа было определено, что требуется отдельная веб-система, которая сохранит наглядность табличного представления, но дополнит его автоматизированной обработкой данных, проверкой конфликтов и более удобным интерфейсом создания бронирований.

**2\. ФОРМИРОВАНИЕ ТРЕБОВАНИЙ К СИСТЕМЕ**

На основе анализа текущего состояния были сформированы требования к разрабатываемому приложению. Система должна обеспечивать просмотр занятости аудиторий, выбор учебной недели, выбор аудитории из списка, отображение временных слотов и создание дополнительных бронирований.

Функциональные требования включают работу с основным расписанием, создание регулярных бронирований по дню недели, паре и типу недели, создание разовых бронирований на конкретную дату и время, а также визуальное различие типов занятости в таблице.

К техническим требованиям были отнесены работа приложения в браузере, использование клиент-серверной архитектуры, обмен данными через REST API, хранение данных в структурированном формате JSON и корректная обработка дат с учетом локального времени.

**3\. ПРОЕКТИРОВАНИЕ АРХИТЕКТУРЫ ПРИЛОЖЕНИЯ**

Для реализации системы была выбрана клиент-серверная архитектура. Клиентская часть отвечает за отображение интерфейса и взаимодействие пользователя с приложением. Серверная часть предоставляет данные и выполняет операции чтения и сохранения информации.

Клиентская часть реализована на React с использованием TypeScript и Vite. Такой стек позволяет строить интерфейс из отдельных компонентов, удобно обновлять данные на странице и поддерживать более строгую структуру кода за счет типизации.

Серверная часть реализована на Node.js и Express. Она обрабатывает HTTP-запросы от клиентского приложения, обращается к JSON-файлам, возвращает данные в формате JSON и сохраняет новые бронирования. Взаимодействие между частями системы выполняется через REST API.

**4\. ОРГАНИЗАЦИЯ ХРАНЕНИЯ ДАННЫХ**

Данные приложения были разделены на несколько JSON-файлов по смысловым группам. Такой подход выбран для реализованной версии системы, так как он позволяет быстро организовать хранение информации без подключения отдельной базы данных.

Для справочных данных используются файлы с аудиториями, преподавателями и временными слотами. Основное расписание хранится отдельно и используется как готовый набор данных, который отображается в интерфейсе, но не редактируется пользователем через приложение.

Созданные бронирования сохраняются в отдельном файле. Это позволяет различать исходное учебное расписание и дополнительные записи, добавленные через форму бронирования. Также предусмотрен отдельный файл для нерабочих дней, который может использоваться при дальнейшем расширении системы.

**5\. РЕАЛИЗАЦИЯ СЕРВЕРНОЙ ЧАСТИ**

Серверная часть была реализована в файле server/index.js. В ней настроено Express-приложение, подключена обработка JSON-запросов и механизм CORS, необходимый для взаимодействия клиентской и серверной частей при разработке.

Через REST API реализованы маршруты получения аудиторий, преподавателей, временных слотов, основного расписания, бронирований и нерабочих дней. Для создания бронирования используется POST-запрос, а для удаления записи - DELETE-запрос по идентификатору.

Серверная часть выполняет роль промежуточного слоя между интерфейсом и файлами данных. Клиентское приложение не обращается к JSON-файлам напрямую, а получает подготовленные ответы от сервера. Это делает структуру системы более понятной и удобной для дальнейшего развития.

**6\. РАЗРАБОТКА КЛИЕНТСКОЙ ЧАСТИ**

Клиентская часть была реализована на React и TypeScript. Основной компонент приложения объединяет остальные элементы интерфейса, хранит выбранную аудиторию, выбранную учебную неделю и загруженные данные.

Интерфейс разделен на несколько компонентов. Компонент выбора недели позволяет менять отображаемый период. Компонент списка аудиторий отвечает за выбор помещения. Таблица занятости отображает дни недели, временные слоты и состояние аудитории. Форма бронирования используется для добавления новых записей.

Такое разделение позволило не перегружать один файл всей логикой интерфейса. Каждый компонент отвечает за отдельную часть пользовательского сценария, а вспомогательные функции для работы с датами и занятостью вынесены в отдельные модули.

**7\. РЕАЛИЗАЦИЯ ТАБЛИЦЫ ЗАНЯТОСТИ**

Одной из центральных частей приложения стала таблица занятости аудиторий. Она строится на основе выбранной недели, выбранной аудитории, временных слотов, основного расписания и списка бронирований.

По строкам таблицы располагаются временные интервалы учебных пар, а по столбцам - дни недели. В каждой ячейке отображается состояние аудитории в конкретный день и конкретное время. Если аудитория занята, ячейка получает визуальное выделение.

В системе различаются занятия из основного расписания, регулярные бронирования и разовые бронирования. Такое разделение помогает пользователю быстро понять, почему аудитория занята, и определить свободные интервалы для нового бронирования.

**8\. СОЗДАНИЕ РЕГУЛЯРНЫХ И РАЗОВЫХ БРОНИРОВАНИЙ**

Форма создания бронирования поддерживает два режима работы. Регулярное бронирование используется для повторяющихся мероприятий, которые проходят по учебному шаблону. Для него указываются аудитория, день недели, номер пары, тип недели, преподаватель и комментарий.

Разовое бронирование создается на конкретную календарную дату и заданный временной интервал. Такой тип записи подходит для экзаменов, пересдач, собраний, консультаций и других мероприятий, которые не повторяются каждую неделю.

После заполнения формы данные отправляются на сервер через REST API. Сервер сохраняет запись в JSON-файле бронирований, после чего клиентская часть обновляет список данных и перестраивает таблицу занятости.

**9\. ПРОВЕРКА КОНФЛИКТОВ И ОБРАБОТКА ДАТ**

Для регулярных бронирований была реализована проверка конфликтов. Система сравнивает новую запись с уже существующими бронированиями по аудитории, дню недели, номеру пары и типу недели. Если найдено пересечение, бронирование не создается.

Особое внимание уделено обработке числителя и знаменателя. Бронирования, созданные только на числитель и только на знаменатель, могут существовать в одном временном слоте. При этом запись на обе недели пересекается с любым из этих вариантов.

Для разовых бронирований используется календарная дата. При создании такой записи система определяет день недели и не допускает бронирование на воскресенье, так как в приложении учебная неделя рассматривается с понедельника по субботу.

**10\. ТЕСТИРОВАНИЕ РАЗРАБОТАННОЙ СИСТЕМЫ**

После реализации основных функций было проведено ручное тестирование приложения. Проверялись запуск серверной части, запуск клиентского приложения, загрузка справочных данных, получение расписания и бронирований.

Также были проверены пользовательские сценарии: выбор аудитории, переключение учебной недели, отображение основного расписания, отображение регулярных и разовых бронирований, создание новых записей и обновление таблицы после сохранения данных.

Отдельно проверялись ограничения системы: запрет разового бронирования на воскресенье и определение конфликта регулярного бронирования. Тестирование показало, что основные сценарии выполняются корректно.

**11\. ПОДГОТОВКА ПРОЕКТНОЙ ДОКУМЕНТАЦИИ**

В ходе работы были подготовлены материалы для описания проектной части ВКР. Сформированы разделы, посвященные ограничениям системы, средствам реализации, структуре проекта, организации хранения данных, серверной и клиентской частям.

Для наглядного описания разработки были подготовлены диаграммы: границы системы, архитектура приложения, файловая структура, логическая модель данных, компонентная структура серверной и клиентской частей, алгоритм определения занятости и пользовательские сценарии.

В приложение к работе были вынесены ключевые фрагменты исходного кода, связанные с обработкой дат, определением занятости, загрузкой данных, проверкой конфликтов и серверным сохранением бронирований.

**12\. РЕЗУЛЬТАТЫ РАБОТЫ**

В результате научно-исследовательской работы была разработана веб-система бронирования аудиторий. Реализован интерфейс для просмотра занятости, выбора недели и аудитории, а также форма создания дополнительных бронирований.

Серверная часть обеспечивает получение данных через REST API и сохранение новых бронирований. Данные системы организованы в виде отдельных JSON-файлов, что позволяет наглядно разделить справочники, расписание, бронирования и нерабочие дни.

Полученное решение закрывает основные проблемы ручного учета в Excel-таблице: упрощает просмотр занятости, снижает риск ошибок при планировании и делает процесс бронирования более понятным и контролируемым.
