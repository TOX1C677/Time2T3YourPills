# План миграции Time2T3YourPills → Flutter (актуальная цель)

**Основа:** `PROJECT_SNAPSHOT_Time2T3YourPills.md`.  
**Продукт:** только **клиентское** приложение для пациента.  
**Не входит в объём:** админка, роль лечащего врача, управление аккаунтами пациентов со стороны врача.

**Инфраструктура:** свой сервер (Ubuntu, белый IP) — **API + серверная БД**; на устройстве — **локальный кэш** для работы без интернета; **при входе** — подтягивание актуальных данных с сервера.

**Дизайн:** отдельно не разработан; требования к адаптации под паркинсонизм — в `DESIGN_GUIDELINES_PARKINSON.md` (чеклист + принципы); визуальная система (Figma) — этап проекта ниже.

## 1. Рекомендация по серверной БД и API

| Компонент | Рекомендация | Зачем |
|-----------|--------------|--------|
| **СУБД** | **PostgreSQL** (LTS на Ubuntu, пакет `postgresql`) | Надёжность, JSON при необходимости, транзакции, индексы по `user_id` / датам приёма, простой бэкап (`pg_dump`). |
| **Доступ из интернета** | **Только API** (за reverse proxy), **не** открывать порт PostgreSQL наружу | Безопасность: БД слушает `localhost` или приватную сеть. |
| **Прокси + TLS** | **Caddy** или **nginx** + Let’s Encrypt на вашем домене/IP | HTTPS обязателен для токенов и данных здоровья. |
| **API** | **REST** (OpenAPI 3) или **JSON-RPC** — на выбор; стек: **FastAPI** (Python), **NestJS** (Node), **Go chi/fiber** | Предсказуемые эндпоинты для Flutter (`dio`). |
| **Кэш/сессии** | Опционально **Redis** только если нужны сессии/очереди; для диплома часто достаточно PostgreSQL + JWT | Упростить деплой. |

**Минимальные сущности на сервере (логически):**

- **Пациент** (профиль): идентификатор учётки, ФИО/отображаемое имя, контакты по желанию, `updated_at`.
- **Препарат:** название, дозировка, локальный/серверный id, связь с пациентом, `updated_at`.
- **План приёма:** правило расписания (например, список слотов времени + дни недели **или** интервал в минутах + окно активности — зафиксировать в ТЗ); связь с препаратом; `updated_at`.

Синхронизация: ответы API с **`updated_at` / версией / etag`** чтобы клиент делал инкрементальное обновление кэша без лишнего трафика.

## 2. Клиент: кэш и синхронизация

**Локальный кэш — не секреты, а снимок данных приложения:** основной KV на диске через **GetStorage** (application documents directory). В коде — абстракция **`KeyValueStore`** и реализация **`GetStorageKeyValueStore`**, плюс **единый список ключей** в `lib/app/storage/storage_keys.dart` (без «магических строк» по проекту).

| Слой | Технология | Роль |
|------|------------|------|
| Локально | **`KeyValueStore`** → GetStorage | Сериализованные снимки: профиль, список препаратов, план приёма, метаданные синка (`last_sync_at` и т.п.) — обычно **JSON в строке** на ключ из `storage_keys.dart`. |
| Сеть | **`dio`** + interceptors (токен, логирование) | HTTP только в **RemoteDataSource**, не в виджетах. |
| Секреты | **`flutter_secure_storage`** (или аналог) | Токены/refresh — **не** в GetStorage. |
| Вход | После успешной аутентификации: **pull** профиля, препаратов, плана → запись в KV **атомарно по ключам** (или один ключ «весь снимок» + парсинг — зафиксировать в ТЗ). | «Обновили при входе». |
| Офлайн | Чтение из `KeyValueStore`; очередь на отправку — **отдельный ключ** (например JSON-массив outbox) | При появлении сети — **push** очереди (или только read-only офлайн на первом этапе — явно зафиксировать в ТЗ). |

**Рекомендация по первой версии:** офлайн — **полное чтение** кэша + редактирование с очередью на отправку; либо офлайн только просмотр и таймер — проще для диплома.

**Замечание:** для объёма данных уровня «десятки препаратов + план» KV достаточно. Если позже понадобятся сложные выборки, индексы и транзакции на устройстве — можно **добавить** SQLite/Drift рядом, не ломая контракт репозитория (другая реализация `LocalDataSource`).

## 3. Архитектура Flutter: экраны не зовут API

**Правило:** виджеты экранов **не** импортируют `dio` и **не** вызывают URL. Вся работа с сетью и БД — через границы слоёв.

```
┌─────────────────────────────────────────┐
│  UI: Screens / Widgets (Stateless/     │
│  ConsumerWidget) — только отображение   │
└─────────────────┬───────────────────────┘
                  │ события / модель состояния
┌─────────────────▼───────────────────────┐
│  Controllers (Riverpod Notifier /       │
│  Bloc/Cubit) — сценарии экрана          │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│  Use cases (опционально, тонкие)        │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│  Repositories — «единая дверь» данных  │
│  (кэш + remote, политика sync)         │
└───────┬─────────────────────┬──────────┘
        │                     │
┌───────▼──────────┐  ┌───────▼──────────┐
│ LocalDataSource  │  │ RemoteDataSource │
│ (`KeyValueStore` │  │ (`dio` + DTO)   │
│  + JSON encode)  │  │                  │
└──────────────────┘  └──────────────────┘
```

- **Controller** (термин в плане = **Notifier / Bloc** в коде): валидирует ввод, вызывает `MedicationRepository.syncAfterLogin()` и т.д., держит `AsyncValue` / состояние загрузки и ошибок для UI.
- **Repository** решает: при старте отдать данные из KV (десериализация), после успешного pull — записать снимок в `KeyValueStore`; при необходимости реактивности UI — обновлять state контроллера / `ref.invalidate` (отдельный `Stream` с диска не обязателен, в отличие от Room/Drift).
- Маппинг JSON ↔ entity — в `data/mappers`, с **unit-тестами**.

Именование папок (пример): `lib/features/medications/presentation/`, `.../application/` (controllers), `.../data/` (sources + repo impl), `lib/core/network/`.

## 5. Тестирование: нужно ли «сразу всё»?

**Краткий ответ:** писать **тесты с первого спринта**, но **не** обязательно 100 % покрытие всех экранов интеграционными тестами сразу. Цель — **стабильность**: регрессии ловятся автоматически, критические сценарии не ломаются молча.

| Тип | Что покрывать с начала | Зачем |
|-----|-------------------------|--------|
| **Unit** | Мапперы DTO, правила расписания, расчёт «следующий приём», логика sync (merge по `updated_at`), валидация полей | Быстро, без эмулятора; ядро безопасности данных. |
| **Widget** | Крупные кнопки, карточка препарата, отображение таймера при разных состояниях | Регресс вёрстки и доступности без полного e2e. |
| **Integration** (`integration_test`) | **2–4 сценария** «золотого пути»: вход (мок API) → список препаратов; старт таймера; офлайн баннер; сохранение профиля | Дороже в поддержке; держать узким набором. |

**CI (с первой недели):** `flutter analyze` + `flutter test` (unit + widget) на каждый push; integration — по расписанию или перед релизом, если тяжело для машины.

**Итог:** да, закладываем **все три уровня**, но **объём** разумный: много unit, выборочно widget, мало но целевых integration. Полное покрытие «каждого экрана» интеграционными тестами обычно **не окупается** на дипломе; лучше расширять integration по мере стабилизации фич.

## 6. Этапы работ (обновлённые)

### Этап 0. Контракты и инфраструктура

- [ ] Спецификация API (OpenAPI): авторизация (например JWT + refresh), эндпоинты профиля / препаратов / плана, модель ошибок.
- [ ] Схема PostgreSQL + миграции на сервере (Flyway/Liquibase или SQL в репо).
- [ ] Репозиторий Flutter: слои + `dio` + **`KeyValueStore` / GetStorage** + `storage_keys.dart`; **без UI**.
- [ ] CI: analyze + test.

**Выход:** поднятый API на Ubuntu (хотя бы в Docker на том же хосте), документ с примером ответов.

### Этап 1. Локальный кэш и репозитории

- [ ] Инициализация GetStorage; контракт **`KeyValueStore`** + **`GetStorageKeyValueStore`**; файл **`lib/app/storage/storage_keys.dart`** (все ключи кэша и префиксы версий при смене формата JSON).
- [ ] `MedicationRepository`, `PatientRepository`, `ScheduleRepository` — чтение/запись снимков через `LocalDataSource` на базе KV; запись после login/pull.
- [ ] Unit-тесты: мапперы JSON ↔ модели, merge синка, **фейковый `KeyValueStore`** в тестах (без диска).

### Этап 2. Контроллеры и сценарий «вход → обновление»

- [ ] Auth + сохранение токена (`flutter_secure_storage`).
- [ ] Controller «приложение после логина»: pull remote → persist → `Stream`/`ref.invalidate`.
- [ ] Экран логина **только** дергает controller, не `dio`.

### Этап 3. Навигация и каркас UI

### Этап 4. Препараты и план (экраны через controllers)

### Этап 5. Таймер и уведомления

## 7. Матрица Android → Flutter (кратко)

| Было (Android) | Станет (Flutter) |
|------------------|------------------|
| Room только локально | **GetStorage** через `KeyValueStore` + JSON-снимки; outbox — отдельный ключ при необходимости |
| Нет сервера | PostgreSQL + собственный REST API |
| ViewModel + Fragment | Controller + Screen |
| Прямые вызовы из UI к данным | Controller → Repository → Local / Remote |
